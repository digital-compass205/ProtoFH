## Executive summary

- **Phase 1 (Python prototype)**: delivered in full and **confirmed perfect in real use** ‚Äî SoupBinTCP connectivity, complete ITCH v2.02 decoding, GLIMPSE snapshot recovery, order books/trades, CLI views, plus an offline exchange simulator. It connected to the exchange, recovered messages, and displayed live market data correctly.
- **Phase 2 (production-shaped C++ system)**: a three-process architecture ‚Äî C++ feed handler (`jnxfh`), C++ in-memory database (`jnxdb`), Python 3.6 web GUI (`jnxweb`) ‚Äî implemented in 9 gated phases (F0‚ÄìF8), one commit per phase, **all complete**.

- **Resilience proven live**: five kill/restart scenarios (FH crash, DB crash, exchange drop, snapshot cold-start, dual crash) all converge to state identical to an uninterrupted run; the testing itself flushed out and fixed one real recovery bug.

## Phase 1 in brief (the foundation Phase 2 stood on)

Phase 1 (plan `JNX_PLAN.md`, tasks T1.1‚ÄìT7.2) built a pure-Python 3.6, stdlib-only prototype: SoupBinTCP client with lossless reconnect-resume, declarative ITCH codec, GLIMPSE snapshot bootstrap, reference data / order-book / trade-tape stores, CLI views, and ‚Äî critically ‚Äî an **exchange simulator** and **golden fixtures** extracted from Japannext's official sample pcaps (222,189 messages, byte-verified against the spec). Three Phase-1 artifacts became Phase 2's safety net: the proven data-handling logic (the porting reference), the simulator (the stand-in test line), and the golden sample (the regression oracle).

## Phase 2: what was built

**Architecture** ‚Äî one exchange message becomes exactly one atomic unit of work end to end:

```
exchange/simulator ‚îÄ‚îÄSoupBinTCP‚îÄ‚îÄ‚ñ∂ jnxfh (C++) ‚îÄ‚îÄUDS records‚îÄ‚îÄ‚ñ∂ jnxdb (C++, 5 tables)
                                      ‚îÇ
                                      ‚îî‚îÄ‚îÄUDP multicast (full-state records)‚îÄ‚îÄ‚ñ∂ jnxweb (Py3.6) ‚îÄ‚îÄ‚ñ∂ browser
```

- **`jnxfh`** owns the live market state (the ported book engine). Per message it applies, builds one ~433-byte UPDATE record carrying the *full* merged ticker state (static + trading state + top-10 book + trade summary + an order-level delta), writes it to `jnxdb` first (DB is authoritative), then multicasts the same bytes.
- **`jnxdb`** is a single-threaded queryable mirror ‚Äî 5 tables keyed by ticker (static, state, order-level book, aggregated book, trades) plus an operator query port. It is also the FH's recovery source: a restarted `jnxfh` downloads the order store and last sequence from `jnxdb` and resumes the exchange session at `last_seq+1` ‚Äî no start-of-day replay. A restarted `jnxdb` is simply re-synced from the FH's memory. No disk persistence anywhere, by design.
- **`jnxweb`** joins the multicast and serves a self-contained web page (hand-rolled HTTP + RFC 6455 WebSocket on one `selectors` loop) showing live book/trades per ticker. Because every record is full-state, packet loss self-heals ‚Äî clients need no recovery protocol.
- The **wire format** is a frozen, versioned contract (`docs/wire_spec.md`), implemented twice (C++ and Python 3.6) and pinned by cross-language golden vectors generated by one side and field-asserted by the other.

**How correctness was enforced with cheap worker models** ‚Äî three mechanical gates, not code review:
1. **Golden vectors** generated by the proven Phase-1 codec pinned every byte of the new C++ codecs (F1).
2. **The parity diff** (F2): canonical state dumps from the prototype and the C++ engine, `diff -r` empty over the full golden day ‚Äî re-run after every later change to the market core.
3. **Scenario invariants** (F6): every crash/restart path must end in state identical to the uninterrupted baseline, with kills triggered by observed progress (SIGKILL, no sleeps). This gate caught a genuine bug in the resync dump (wrongly-keyed rows for tickers whose orders had all closed) ‚Äî exactly the class of defect that survives unit tests.

**Environment discipline.** The dev container (Ubuntu, gcc 15, Python 3.14, no docker) is nothing like the target (RHEL 8.10, gcc 8.5, Python 3.6, no third-party packages). Countermeasures: `-std=c++11 -Wall -Wextra -Werror -pedantic` on every compile; a purpose-built AST linter (`tools/py36check.py`) gating all production Python; and a `dist` tarball proven to build with only gcc and make. Final assurance still requires the `TARGET_VALIDATION.md` checklist on a real target box.

**Deliverables inventory**: `cpp/` (‚âà itch/soup/wire codecs, market core, db, fh, 65 tests, strict Makefile with `bench`/`test-asan`/`dist`), `jnxweb/`, `tools/` (dbquery, mcast_spy, ws_probe, compare_db_dump, run_e2e with 6 scenarios, py36check, dump/vector generators), docs (`wire_spec.md`, `OPERATIONS.md`, `RECOVERY.md`, `TARGET_VALIDATION.md`, sample configs), and the dist tarball.

## Observations on the AI-agent process

- The **plan-as-contract** approach worked: each phase prompt embedded the ground rules, the protocol cheat sheet, exact verification commands, and known pitfalls; Sonnet agents then rarely needed correction. The two intervention types that mattered were environmental (session-limit interruptions ‚Äî resumed cleanly from transcripts) and architectural (deciding gates, freezing formats, folding discovered semantics back into the plan).
- **Independent re-verification before every commit** caught nothing the agents' own runs hadn't ‚Äî but it is what made their green claims trustworthy, and it is cheap.
- Fresh agents with complete prompts beat long-lived agents once transcripts grew: F6‚ÄìF8 each ran cold, faster and with no context tax.
- Cost profile: five Sonnet agents, ~1.5M subagent tokens total, over roughly two days of elapsed sessions for all of Phase 2.

## Known limitations / follow-ups

1. `jnxweb` imports the prototype's `jnxfeed.net.reactor` (the dist ships a shim) ‚Äî should be made self-contained.
2. The simulator renumbers sequences from 1 (no offset option), so integration assertions are count-based rather than using original pcap sequence numbers.
3. Order-number collisions across order-book groups remain an open exchange question (Q1 in `CONNECTIVITY.md`); the code counts and warns.
4. The full-pipeline benchmark (~38k msg/s) is bounded by the Python simulator, not the C++ (decode+apply ceiling ~630k msg/s); a C++ feeder would be needed to measure the true pipeline limit.
5. Target-box validation and Japannext UAT are the only steps between this state and production readiness ‚Äî see `TARGET_VALIDATION.md`.
